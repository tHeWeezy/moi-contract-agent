## 1. 需求背景与目标

- **背景：** 目前用户在查询项目进度或客户档案时，需要进入繁琐的层级菜单并填写筛选表单。为了提升检索效率，计划在“合同智能体”中引入“信息问询”功能，允许用户通过自然语言直接查询。
    
- **目标：**
    
    1. 实现“信息问询”专属对话框的 UI 布局与基础交互。
        
    2. 在无真实 AI 模型介入的阶段，通过前端预设关键词和假数据（Mock Data），跑通“文本输入 -> 意图识别 -> 卡片渲染 -> 详情页跳转”的完整闭环。
        
    3. 无缝衔接已开发完成的详情页（`moi_c_ProjectInfo` 和 `moi_c_ClientInfo`）。
        
---

## 2. 核心功能范围 (Scope)

- **包含：** 底部输入组件设计、对话气泡展示、结构化信息卡片（项目卡片/客户卡片）渲染、基于特定关键词的 Mock 回复逻辑、卡片按钮跳转逻辑。
    
- **不包含：** 复杂的 NLP（自然语言处理）意图识别、上下文多轮对话记忆、真实的后端数据库查询接口接入（将在此版本验证通过后作为二期需求）。
    

---

## 3. 详细功能说明

### 3.1 前端 UI 与入口交互

- **页面布局：** 采用标准的左右分栏聊天室布局（参考设计图）。右侧为主干对话区，底部为输入区。
    
- **问询模式切换：** * 输入框左侧需固定展示“信息问询 ∨”的下拉标签。
    
    - **交互说明：** 选定此模式后，占位符 (Placeholder) 提示语应变更为：“请输入您的查询条件，例如：查询某项目的进度”。
        
- **消息发送：** 支持点击“发送”按钮或键盘回车 (`Enter`) 发送消息。发送时输入框内容清空，消息以上推形式进入聊天流。
    

### 3.2 假数据 (Mock) 识别逻辑

为了在没有后端大模型的情况下跑通流程，前端需在用户发送消息后，拦截文本并进行简单的关键词匹配（Switch/Case 逻辑）：

- **场景 A：项目查询意图**
    
    - **触发条件：** 用户输入的文本中包含关键词 **“项目”**（例：“查一下最近的项目”）。
        
    - **系统响应：** 延迟 800ms（模拟网络请求），回复引导语，并下发【项目信息卡片】的假数据。
        
- **场景 B：客户查询意图**
    
    - **触发条件：** 用户输入的文本中包含关键词 **“客户”**（例：“帮我找个客户”）。
        
    - **系统响应：** 延迟 800ms，回复引导语，并下发【客户信息卡片】的假数据。
        
- **场景 C：兜底回复 (Fallback)**
    
    - **触发条件：** 输入内容既无“项目”也无“客户”。
        
    - **系统响应：** 回复兜底文案：“抱歉，测试阶段我仅支持通过包含‘项目’或‘客户’的关键词进行模拟查询，请重新输入。”
        

### 3.3 结构化数据展示 (整合数据明细卡片)

当 AI 命中查询意图并返回多条相关数据时，采用“结论文本 + 整合数据卡片”的组合形式。卡片内嵌轻量级表格，并取消行级操作，将跳转动作收敛至卡片全局入口。

#### 3.3.1 组件整体构成

1. **结论文本 (Summary Text)：** 位于卡片上方，AI 的自然语言摘要，告知用户查询结果的实体及数量。
    
2. **整合数据卡片 (Data Card)：**
    
    - **卡片头部 (Card Header)：** 左侧展示明细标题及总数（例：项目明细 共 2 份）；右侧放置唯一的**全局跳转按钮**（例：👁 查看项目详情）。
        
    - **卡片主体 (Card Body - Table)：** 纯展示型表格，无操作列。
        
    - **卡片底部 (Card Footer - 可选)：** 展示数据来源标签（例：来源：Matrix Origin 客户关系管理系统）。
        

#### 3.3.2 项目信息查询结果 (Mock 场景 A)

- **结论文本：** “为您找到与该关键词相关的 2 个项目信息。”
    
- **卡片头部：**
    
    - 左侧标题：`项目明细（共 2 份）`
        
    - 右侧按钮：`👁 查看项目详情` （点击携带 Mock `ProjectID` 跳转至 `moi_c_ProjectInfo` 页面）
        
- **表格表头定义 (Columns)：**
    
    - `所属项目编号` (Project ID) - _例：P-20260316_
        
    - `项目名称` (Project Name) - _例：数据库二期扩容项目_
        
    - `当前状态` (Status) - _例：执行中_
        
    - `负责人` (Manager) - _例：张三_
        
    - _(注：移除原有的“操作”列)_
        

#### 3.3.3 客户信息查询结果 (Mock 场景 B)

- **结论文本：** “为您匹配到「XXX」相关的 3 份客户档案记录。”
    
- **卡片头部：**
    
    - 左侧标题：`客户明细（共 3 份）`
        
    - 右侧按钮：`👁 查看客户档案` （点击携带 Mock `ClientID` 跳转至 `moi_c_ClientInfo` 页面）
        
- **表格表头定义 (Columns)：**
    
    - `客户编号` (Client ID) - _例：C-884821_
        
    - `客户名称` (Client Name) - _例：广联达科技股份有限公司_
        
    - `联系人` (Contact) - _例：李总_
        
    - `客户级别` (Level) - _例：核心 VIP_
        
    - _(注：移除原有的“操作”列)_
        

#### 3.3.4 跳转逻辑与边界规则 (动态路由)

为了兼容“单实体精确查询”与“多实体泛查询”的不同场景，整合数据卡片右上角的**全局跳转按钮**需具备基于数据结果的动态判断逻辑：

- **场景一：单实体命中（精确查询）**
    
    - **触发条件：** 查询结果表格内的所有数据，均归属于同一个项目（唯一的 `ProjectID`）或同一个客户（唯一的 `ClientID`）。例如：查询“郑州地铁集团”名下的所有订单。
        
    - **按钮文案：** `👁 查看项目详情` 或 `👁 查看客户档案`。
        
    - **跳转动作：** 直接携带该唯一的 ID 跳转至对应的详情页（`moi_c_ProjectInfo` 或 `moi_c_ClientInfo`）。
        
- **场景二：多实体命中（泛查询）**
    
    - **触发条件：** 查询结果表格内的数据，包含多个不同的 `ProjectID` 或 `ClientID`。例如：查询“本月新增的大客户”，结果包含 A、B、C 三家不同公司。
        
    - **按钮文案：** `👁 查看完整列表`。
        
    - **跳转动作：** 携带用户的**搜索关键词**（或命中数据的 ID 集合参数），跳转至对应的**系统原生列表页**（如：客户列表大厅/项目列表大厅）。列表页需自动应用该过滤条件，展示这几家公司的列表，随后由用户在列表页中自行点击进入特定详情。
        
- **表格视觉边界：** 卡片内嵌的表格需设定最大高度（建议展示 5 行数据的高度）。当结果超过 5 行时，表格内部触发垂直滚动条（Scrollbar），严格防止聊天气泡高度无限拉长，影响上下文阅读
### 3.4 路由与跳转逻辑 (核心闭环)

此部分是打通现有系统的关键：

- **跳转动作：** 用户点击卡片底部的操作按钮时，触发页面跳转/路由切换。
    
- **参数携带：** * 点击【项目卡片】，需携带该 Mock 数据的 `ProjectID` 跳转至目标文件夹：**`moi_c_ProjectInfo`**中的项目详情页面。
    
    - 点击【客户卡片】，需携带该 Mock 数据的 `ClientID` 跳转至目标文件夹：**`moi_c_ClientInfo`**中的客户详情页面。
        
- **展示形式：** 跳转通过新开浏览器页签。

## 4. 埋点与非功能性需求

- **加载状态：** 用户发送消息后，至卡片渲染出来之前，需要有简单的 Loading 动画（如“正在查询中...”或三个跳动的小圆点），以缓解等待焦虑。
    
- **扩展性准备：** 前端在定义数据结构时，必须将“聊天流对象”设计为标准格式（包含 `id`, `role`, `content`, `cardType`, `cardData` 等字段），确保未来真实接入 AI 接口时，只需替换数据源，无需重构 UI 组件。
